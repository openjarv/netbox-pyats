# Changelog

All notable changes to netbox-pyats are documented here. The format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project
adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [0.1.0] - Unreleased

### Added

- Secret/PII leakage gate (ATW-116): committed `.pre-commit-config.yaml` wiring gitleaks with a custom `.gitleaks.toml` allowlist/regex set covering Tailscale CGNAT IPs (`100.64.0.0/10`), Tailscale DNS (`*.ts.net` / `*.tscale.net` / `*.tailscale.com`), Paperclip agent/server identifiers, private keys, and the upstream gitleaks defaults. The `<...>` placeholder convention used in `docs/developer/remote-access.md` is allowlisted so the redacted runbook does not false-positive. Pure-Python regression test in `netbox_pyats/tests/test_secret_detection.py`.
- NetBox plugin scaffold (`netbox_pyats/`): `PluginConfig`, `PLUGINS_CONFIG` schema, entry point, navigation menu (PyATS Credentials, Add Credential).
- `PyatsCredential` model with Fernet field-level encryption (`netbox_pyats.crypto`): `password` and `enable_secret` stored as ciphertext; plaintext only lives in-memory on the pyATS `Testbed` built by the worker. Key from `PLUGINS_CONFIG['netbox_pyats']['credential_key']` (recommended) with a documented `SECRET_KEY`-derived dev fallback (warns).
- `PyatsCredential` CRUD views + templates, REST API viewset (write-only secrets — never returned), GraphQL type (ciphertext fields excluded), search index, filterset, table.
- `build_testbed(device_qs)` bridge: materializes a `pyats.topology.Testbed` from a NetBox Device queryset — maps Platform slug → pyATS `os`, resolves mgmt IP from `primary_ip4`/`primary_ip6`, attaches the device's `PyatsCredential`, and **flags unsupported platforms gracefully** (`os = "unsupported - no parser"`) instead of raising. `on_unsupported="flag"` (default) vs `"skip"`.
- `TestbedBuildReport` summary (supported/unsupported counts + reasons) so callers can surface a UI summary without re-iterating the testbed.
- `PyatsSnapshot` model (Phase 2, ATW-13): JSONB `data` + `parser_warnings`, `kind` (config/state/full), `status` (success/unsupported/error), `triggered_by`, `genie_version`/`pyats_version`, `size_bytes`. Indexed by `(device, -captured_at)` and `(device, kind, -captured_at)` for the device-page history and the diff/compliance pickers.
- `capture_snapshot` RQ job (`netbox_pyats.jobs`): builds a one-device testbed, connects via Unicon, runs parser-based config capture (`device.parse('show running-config')`) and/or state capture (a small OS-agnostic command set via `device.parse(...)`, see `STATE_COMMANDS` in `netbox_pyats.capture`), serializes to JSONB, and stores a `PyatsSnapshot` row. Per-command parser misses are recorded as warnings; unsupported platforms short-circuit to an `unsupported` row; capture errors are recorded as `error` rows with the exception text in `parser_warnings` — the row is always created so the device-page history shows the outcome.
- Dedicated `pyats` RQ queue (declared via `NetBoxPyATSConfig.queues`) + worker Dockerfile (`dev/Dockerfile.pyats-worker`) + dev compose service (`netbox-pyats-worker`), isolating pyATS/Genie work from NetBox's default workers. Worker deployment guide in `docs/workers.md`.
- Device-page "PyATS" tab (`PluginTemplateExtension`): capture button (config / state / full) + recent-snapshot history with status badges and a warnings indicator. Unsupported platforms surface a banner before the operator clicks.
- `PyatsSnapshot` list/detail views + templates (JSONB rendered server-side via the `|json` filter), REST API viewset (read-only in v1), GraphQL type, search index, filterset, table.
- `PyatsSnapshotDiff` model (Phase 3, ATW-14): JSONB `diff` (structured tree) + `summary` (added/removed/changed/unchanged counts) + `parser_warnings`, `status` (success/empty/error), `size_bytes`. FKs to `device`, `before` snapshot, `after` snapshot. Indexed by `(device, -created)` and `(device, status, -created)`.
- `diff_snapshots` pure-Python diff engine (`netbox_pyats.diff`): recursive structured diff over the two snapshots' JSONB `data` dicts — added/removed/changed/unchanged leaves, nested dicts, positional lists. JSON-serializable end-to-end, deterministic, testable without Genie installed. Graceful degradation: empty inputs → `status="empty"`, malformed inputs → `status="error"` with a warning.
- `run_diff` RQ job (`netbox_pyats.jobs`): loads two `PyatsSnapshot` rows of the same device, runs `diff_snapshots`, persists a `PyatsSnapshotDiff` row. Enqueued on the dedicated `pyats` queue via `enqueue_diff` (the diff engine itself needs no pyATS, but the queue is shared for isolation and a single worker image). Same-device invariant enforced; missing-snapshot and device-mismatch paths still write an error row.
- Diff viewer view + template (`/plugins/pyats/diffs/<pk>/`): server-rendered collapsible `<details>` tree (no JS) with before/after values for changed leaves, summary badges, raw-JSON fallback, and parser warnings.
- Device-page "PyATS" tab diff picker: select two snapshots of the same device → enqueues `run_diff`. Only offered when the device has ≥2 snapshots. Recent-diffs list with status badges and before→after links.
- `PyatsSnapshotDiff` list/detail/delete/bulk-delete views + REST API viewset (read-only in v1) + GraphQL type + search index + filterset + table + nav menu entry.
- `PyatsGoldenConfig` model (Phase 4, ATW-15): an operator-authored "expected" running-config for a NetBox device. `config_text` is free text (typed/pasted or promoted from a known-good snapshot via the `source` field — `manual` vs. `snapshot` — with a `source_snapshot` FK for provenance). Multiple goldens per device allowed (e.g. `baseline`, `post-maintenance-window`). Fully editable via REST in v1 so operators can seed goldens from an external config-management tool.
- `run_compliance` pure-Python compliance engine (`netbox_pyats.compliance`): classifies a golden config text vs. a captured snapshot's raw `data["config_raw"]` running-config text as `compliant` (no added/removed lines), `drift` (any divergence), or `error` (empty golden, snapshot with no `config_raw` payload, or `unsupported`/`error` snapshot). v1 is **line-oriented text diff** (line-set diff, order-independent; trailing whitespace stripped, blank/`!` delimiter lines dropped as noise) — the diff tree has the same JSON-serializable shape as `PyatsSnapshotDiff.diff` so the Phase 3 `inc/diff_tree.html` partial renders it unchanged (each leaf is a config line marked `unchanged` / `added` / `removed`). NetBox/RQ/Genie-free at the engine layer so it is unit-testable without a device. Ordered/structured compliance (e.g. ACL order, interface MTU constraints) is deferred to v2, where the golden can be parsed with the same Genie parser the snapshot used (requiring a device connection or a parser-only harness).
- `run_compliance` RQ job (`netbox_pyats.jobs`): loads a `PyatsGoldenConfig` and a `PyatsSnapshot` for the same device, extracts the golden `config_text` and the snapshot's raw `data["config_raw"]` (with a legacy `data["config"]["raw"]` fallback for snapshots captured before `config_raw` was added), runs `run_compliance`, persists a `PyatsComplianceRun` row. Enqueued on the dedicated `pyats` queue via `enqueue_compliance`. The job itself needs no pyATS (operates on persisted text) but runs on `pyats` for isolation and a single worker image. Same-device invariant enforced; missing-golden, missing-snapshot, and device-mismatch paths still write an error row (golden/snapshot FKs are nullable `on_delete=SET_NULL` for the dangling-FK error-row path — see migration 0006).
- `PyatsComplianceRun` model (Phase 4, ATW-15): JSONB `diff` (structured tree, same shape as `PyatsSnapshotDiff.diff`) + `summary` (added/removed/changed/unchanged counts; `changed` is always 0 for the line-set diff) + `parser_warnings`, `result` (compliant/drift/error), `size_bytes`. FKs to `device`, `golden` (nullable, `SET_NULL`), `snapshot` (nullable, `SET_NULL`). Indexed by `(device, -created)` and `(device, result, -created)`.
- Compliance-run viewer view + template (`/plugins/pyats/compliance-runs/<pk>/`): reuses the Phase 3 diff-tree partial so the same server-rendered collapsible before/after tree renders the golden-vs-snapshot divergence, with a result badge (compliant / drift / error), drift indicator, and any warnings.
- Device-page "PyATS" tab compliance picker: select a golden config and a config/full snapshot → enqueues `run_compliance`. Only offered when the device has ≥1 golden config and ≥1 config/full snapshot. Recent-compliance-runs list with result badges and golden-vs-snapshot links.
- `PyatsGoldenConfig` list/detail/edit/delete/bulk-delete views + REST API viewset (fully editable in v1) + search index + filterset + table + nav menu entry. `PyatsComplianceRun` list/detail/delete/bulk-delete views + REST API viewset (read-only in v1) + search index + filterset + table + nav menu entry. (GraphQL types for these two Phase 4 models are deferred — `netbox_pyats/graphql/schema.py` ships types for `PyatsCredential`, `PyatsSnapshot`, and `PyatsSnapshotDiff` only.)
- `0005_compliance_engine` migration adding the `PyatsGoldenConfig` and `PyatsComplianceRun` tables.
- `0006_compliance_run_nullable_fks` migration making `PyatsComplianceRun.golden` and `.snapshot` nullable (`on_delete=SET_NULL`) so the compliance job's dangling-FK error-row path can persist (records the missing id in `parser_warnings` rather than failing `full_clean()`). Mirrors the Phase 3 `PyatsSnapshotDiff.before`/`.after` nullability.
- `0007_snapshot_parsed_os` migration adding `PyatsSnapshot.parsed_os` (CharField, blank/default `""`) carrying the pyATS os string used by the capture (e.g. `iosxe`/`iosxr`/`nxos`) so v2 structured compliance can pick the right Genie parser for a snapshot whose device has since been deleted. Additive only; old snapshots keep `parsed_os=""` and remain usable on the v1 raw-text compliance path. Added to `clone_fields` so cloned snapshots carry it.
- Capture now stores `data["config_raw"]` (the raw `show running-config` text) alongside `data["config"]` (the Genie structured dict) on config/full snapshots, for the Phase 4 compliance engine's line-oriented text diff. Legacy snapshots (pre-0006) fall back to `data["config"]["raw"]`.
- Capture now records the device's pyATS os string on `CaptureResult.parsed_os` and persists it to `PyatsSnapshot.parsed_os` on both the success and error paths (unsupported captures leave it blank — `UNSUPPORTED_OS` is not a real os string). Provenance for v2 structured compliance.
- Unit tests: compliance engine covering compliant/drift/error classification, empty/unsupported/malformed inputs, JSON-serializability, realistic golden-vs-snapshot raw-text payloads, and the end-to-end compliance path (golden text → snapshot raw text) that was missing in the original v1; compliance models, views, and REST API tests; snapshot-raw extraction (config_raw + legacy fallback). NetBox-dependent compliance tests skip cleanly when NetBox is absent.
- Unit tests: crypto round-trip + key resolution + rotation contract; testbed builder covering platform mapping, mgmt IP resolution, credential attachment, unsupported-flag vs skip, report summary; capture logic covering unsupported/config/state/full/error paths with a stubbed genie; diff engine covering added/removed/changed/unchanged, nested dicts, positional lists, empty/error inputs, JSON-serializability, realistic snapshot payloads. NetBox-dependent model/view/API/snapshot/diff tests skip cleanly when NetBox is absent.
- Local dev environment via `docker-compose.dev.yml` (NetBox 4.6 + PostgreSQL + Redis + dedicated pyats worker).
- CI workflow (`.github/workflows/ci.yml`, GitHub Actions): three lanes — `lint` (black/isort/flake8), `unit` (pure-Python tests on Python 3.10/3.11/3.12 with `pyats[full]` installed so the testbed suite runs), and `integration` (full NetBox-dependent suite inside the dev container). The integration lane is gating (`continue-on-error: false`); the NetBox 4.6 dev-image compatibility work (ATW-25) and the gating flip (ATW-49) have landed.

### Fixed

- Dropped bogus `("dcim", "0050_custom_field_choice_set_remove")` dependency from all three plugin migrations (`0001_initial`, `0002_pyatssnapshot`, `0003_pyatssnapshotdiff`). That dcim migration does not exist in any released NetBox 4.6.x image and raised `NodeNotFoundError` on `manage.py migrate netbox_pyats`. Initial migration now uses `dependencies = []`; later migrations depend only on the prior `netbox_pyats` migration — the NetBox reference-plugin convention (no coupling to NetBox's dcim migration history, which NetBox periodically squashes). See ATW-25 and [ADR-0003](docs/adr/0003-netbox46-migration-and-worker-toolchain.md).
- Added `0004_reconcile_netboxmodel_fields` migration to reconcile the three plugin models with NetBox's `NetBoxModel` base classes, so `manage.py makemigrations netbox_pyats --check --dry-run` reports `No changes` inside NetBox 4.6. The initial migrations (0001-0003) were authored outside a NetBox context and were missing attributes NetBox adds to the inherited `created` / `last_updated` / `custom_field_data` fields (`null=True`, `CustomFieldJSONEncoder`). They also declared `tags` as a plain `ManyToManyField` (auto through-table `netbox_pyats_<model>_tags`) instead of NetBox's `TaggableManager` backed by `extras.TaggedItem`; the new migration swaps the field via `RemoveField`+`AddField` (an `AlterField` cannot change M2M to `TaggableManager`). The Django 5.x cosmetic index-rename suggestions are intentionally NOT applied — the explicit `name=` on the original indexes (now mirrored on the model `Meta`) is the durable form and suppresses the rename suggestions. No model behavior changes. See ATW-32.
- `0008_pyatssnapshotdiff_nullable_fks` migration making `PyatsSnapshotDiff.before` and `.after` nullable (`on_delete=SET_NULL`) so the diff job's `DoesNotExist` error-row path can persist when a snapshot is deleted between the user clicking "Diff" and the worker picking up the job. Previously both FKs were `on_delete=CASCADE` / non-nullable (as shipped in `0003_pyatssnapshotdiff`), so `full_clean()` rejected the error row on a dangling FK and the original `DoesNotExist` re-raised — the operator only saw a failed NetBox Job row, never the intended per-diff error row with the missing ids in `parser_warnings`. Same latent bug ATW-62 blocker 3 filed for Phase 4 and PR #20 fixed via `0006_compliance_run_nullable_fks`; Phase 3 has had it since `0003` shipped. Diff job's `DoesNotExist` branch now sets `before=None`/`after=None` and records `before_id`/`after_id` in `parser_warnings` (mirroring the compliance job's pattern). `PyatsSnapshotDiff.__str__` and the detail/list templates are now null-safe. The Phase 4 compliance PR's "matches Phase 3 nullability" docstring claim (migration `0006`, `PyatsComplianceRun` model, `run_compliance_job` docstring) is now true retroactively and forward-points to this migration. Regression tests in `test_snapshots.py` cover both the model-level null-FK round-trip and the `run_diff_job` `DoesNotExist` branch end-to-end. See ATW-68.
- Job-callable error-row write-failure path: when the inner `except` in `capture_snapshot_job` / `run_diff_job` / `run_compliance_job` fails to persist the ADR-0002 error-row (e.g. `full_clean()` rejects it), the reference to the unsaved row is now dropped so the outer `except` records a `PyatsJob` error rather than trying to FK-link an unsaved row (which raised `save() prohibited to prevent data loss due to unsaved related object`). The job still re-raises so `core.Job` is marked failed. Regression test: `test_pyatsjob.py::CaptureJobPyatsJobPlumbingTest::test_error_when_result_row_write_fails_records_error_text`.
- `PyatsJob.started_at` / `finished_at` now use `django.utils.timezone.now()` instead of `datetime.now()` so the values are timezone-aware (Django's `USE_TZ=True` runtime warning on naive datetimes is gone).

### Added

- `PyatsJob` model (Phase 5, ATW-16): a plugin-scoped job-tracking row that bridges NetBox's `core.models.Job` (the RQ-level tracking row) and the plugin's result rows (`PyatsSnapshot`, `PyatsSnapshotDiff`, `PyatsComplianceRun`). Each `enqueue_*` helper creates a `PyatsJob(status=pending)` before `Job.enqueue`; the job callable sets `running` at entry, `success` (with the result-row FK) on a clean run (including `unsupported`/`error` *result rows* — those are successful *jobs* per ADR-0002), or `error` (with the exception text in the `error` field) when the result row could not be written. `summary` JSONField carries batch counts (`{supported, unsupported, errored, total}`) and `partial` status for batches that did not crash but had per-device failures. Structural contract: [ADR-0005](docs/adr/0005-pyatsjob-model.md). No backfill — pre-ATW-16 jobs do not get retroactive `PyatsJob` rows; the unified jobs view starts populated from ATW-16 forward.
- `PyatsJobTypeChoices` (`capture | diff | compliance | batch_capture`) + `PyatsJobStatusChoices` (`pending | running | success | error | partial`) choice sets. `partial` and `pending`/`running` extend ADR-0002's status-vocabulary table (ADR-0005 §2 is the follow-up ADR that extends it). The per-result-row statuses (`PyatsSnapshot.status`, `PyatsSnapshotDiff.status`, `PyatsComplianceRun.result`) are unchanged.
- `enqueue_batch_capture(devices_qs, *, kind, user)` + `batch_capture_job` RQ callable (Phase 5, ATW-16): the first multi-device op. Iterates devices, reuses `capture_snapshot_for_netbox_device`, counts `supported`/`unsupported`/`errored`, persists a `PyatsSnapshot` row per successful device, and sets the `PyatsJob` to `success` (all clean) or `partial` (any per-device failure or unsupported platform). A device deleted between enqueue and run is silently skipped (the summary reflects what was actually iterated). Batch diff/compliance is out of scope for v1.
- Unified PyATS Jobs view (`/plugins/pyats/jobs/`): `PyatsJobListView` (filterable by `job_type` / `status` / `device`) + `PyatsJobView` (detail: links to `core_job` + result row + `error` in `<pre>` + batch `summary`). Jobs are append-only history — no edit view; standard delete only. Navigation entry "PyATS Jobs".
- Device-list bulk "PyATS capture" action: `DeviceBulkCaptureView` (plain `View` with `PermissionRequiredMixin`, requires `netbox_pyats.add_pyatsjob`) enqueues a single `enqueue_batch_capture` job targeting the selected device set. Wired at `/plugins/pyats/devices/bulk-capture/`.
- Supported-platforms report (`/plugins/pyats/supported-platforms/`, Phase 5, ATW-16, Option A): `SupportedPlatformsReportView` renders the static `PLATFORM_SLUG_TO_PYATS_OS` map + per-platform NetBox device counts + an "unsupported" summary. Web-process-safe — reads a plain Python dict, no Genie import (ADR-0001 §6). Live Genie introspection (Option B) is v2. Navigation entry "Supported Platforms".
- `0009_pyatsjob` migration adding the `PyatsJob` table (linear after `0008`, no backfill). Indexed by `(job_type, -created)`, `(status, -created)`, and `(device, -created)` for the unified jobs view's filter axes.
- `PyatsJob` REST API viewset (read-only in v1, `http_method_names = ["get","head","options"]`) + GraphQL type + search index + filterset + table + filter form.
- ADR-0005 (`docs/adr/0005-pyatsjob-model.md`) accepted: PyatsJob unified job-tracking model + status vocabulary extension.
- Tests: `test_pyatsjob.py` (model persistence, `related_result` accessor, status color map, ADR-0005 §3 plumbing for capture/diff/compliance — success / unsupported-result-row / error-row-write-failure / DoesNotExist / batch summary + `partial` status, deleted-device-skip) and `test_supported_platforms.py` (static-map invariants, web-process-safety, rendered report contents).
